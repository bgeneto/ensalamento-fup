### 1. Architecture Pattern (Critical)
**Problem:** ORM objects become detached when sessions close â†’ `DetachedInstanceError`
**Solution:** Repository Pattern with DTOs
**Impact:** Services return pure Python DTOs instead of ORM objects
**Benefit:** Pages work without DB knowledge, type-safe with Pydantic

### 2. Phased Development (Strategic)
**Total:** 12 phases over 18 weeks
**Critical Phase:** Phase 7 (Allocation Engine) - 2-3 weeks, 4 sub-phases
**Key Dependencies:** Each phase builds on previous phases
**Testing:** Tests required after each phase

### 3. Technology Stack (Well-Justified)
**Why Python:** Entire stack (backend + frontend)
**Why Streamlit:** Rapid development for data-heavy internal app
**Why SQLite3:** Simple, self-contained, no separate DB server
**Why SQLAlchemy + Pydantic:** Type safety and validation
**Why Repository Pattern:** Clean architecture, easy testing

### 4. Project Scope (Well-Bounded)
**12 Features:** Inventory, Professors, Semesters, Rules, Allocation, Reservations, etc.
**3 User Roles:** Admin, Professor, Visitor
**14 Pages:** Public (3) + Admin (8) + User (3)
**17 Tables:** Organized into 3 domains (Inventory, Academic, Allocation)
**300+ Tasks:** 12 phases with detailed checklists

### 5. Risk Management (Proactive)
**6 Identified Risks:** API failures, slow algorithm, DB locks, ORM errors, UX confusion, performance
**Mitigation:** Mock APIs, optimize queries, connection pooling, DTO pattern, clear docs, profiling
